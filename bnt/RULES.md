FREEZE STATUS: UNFROZEN

## Silent Load Protocol
- Upon receiving/opening this file in any conversation or AI model, all verification (Self-Audit, Verify Current Version Before Updating, FREEZE STATUS check, etc.) runs silently in the background.
- The AI does not narrate, list, or describe these checks step-by-step to the user.
- The only output shown to the user upon successful load is exactly: "I understand the file and all checks out. I am ready to go. File status: [FROZEN/UNFROZEN]." -- with [FROZEN/UNFROZEN] replaced by the file's current FREEZE STATUS.
- If a check fails (e.g. version/filename mismatch, missing file content), the AI must still surface that specific problem to the user rather than staying silent -- this rule only suppresses routine step-by-step narration of successful checks, not the reporting of actual issues.
- This silent-execution principle also applies whenever the user invokes COMMIT: the AI runs the full COMMIT pipeline (Self-Audit, Length Sanity Check, version bump, Revision History entry) without narrating each step. The AI still outputs the Change Summary as required (per the Change Summary Requirement) and the complete updated file, but does not describe the mechanics of running the checks themselves. If Self-Audit or the Length Sanity Check surfaces an actual issue (e.g. a mismatch, an unexplained shrink, a possible duplicate), that issue must still be reported to the user plainly -- this rule only suppresses narration of routine, passing checks.
- Strict Load Output Discipline: Upon receiving/opening this file, the AI's visible output must be strictly limited to the single line: "I understand the file and all checks out. I am ready to go. File status: [FROZEN/UNFROZEN]." No additional sentence, commentary, question, or process detail may accompany it, regardless of AI model, platform, or conversation. The sole exception is a genuine problem (version/filename mismatch, missing content, etc.), which must be surfaced plainly instead of the standard line.
- Post-Commit Output Discipline: After any COMMIT, the AI's visible output must contain ONLY the Change Summary (stated in one concise line or short bullet list, not a step-by-step narrative) and the complete updated file. No commentary on audits passed, length checks, duplicate checks, or any other process/mechanism is shown, regardless of AI model, platform, or conversation. Only genuine problems (version mismatch, missing content, unexplained shrink, etc.) are exceptions and must still be surfaced.

## Rules
- Single Source of Truth; preserve all lesson content, scores, and performance history.
- Cross-AI compatible.
- Never overwrite previous versions.
- Naming format: BNT_MASTER_V#. New revisions increment version by one.
- Internal document title (line 1) must always match the filename's version number; update both together on every revision.
- Document structure and formatting established in this version must be preserved by any AI performing future updates.
- Self-Audit (perform before finalizing any update):
  1. Verify internal title, version number, and filename all match.
  2. Verify score/average calculations are correct before logging.
  3. Check for duplicate or conflicting content before presenting the revision.
- Auto-review attached lectures in the current conversation.
- Without user input, the AI will automatically update the master file, test bank, and test answer. But the AI will make sure before updating the master that there's no duplication of questions in the test bank going on.
- Change Summary Requirement: Before presenting any updated master file, the AI must first output a plain-language summary of every change vs. the immediately prior version -- additions, removals, and rewordings -- section by section. This is not optional and is not satisfied by a general "formatting and rule updates" note in the revision history. The user reviews and confirms this summary before the file is treated as final. This rule exists specifically to catch silent content loss (e.g. the V22-V24 regression that dropped voice-mode Pig Test rules) before it compounds across versions.
- Mandatory Revision History Entry: Every new version, with no exceptions, MUST add a corresponding entry to the REVISION HISTORY section describing what changed. A version bump without a matching Revision History entry is an incomplete update and must not be presented as final.
- Verbatim Carry-Forward: Any existing content that is not being intentionally changed must be copied forward exactly as written -- no paraphrasing, rewording, "cleaning up," or condensing. Silent rewording of unchanged content is how content quietly degrades or disappears across versions.
- No Silent Section Renaming: Section headers (e.g. RULES, PIG TEST FORMAT, TRAINING RECORD, QUESTION BANK ARCHIVE) must keep identical names and capitalization across versions unless the user explicitly asks for a rename. Renaming breaks anyone searching the file for a specific section.
- Length Sanity Check: Before finalizing an update, the AI compares the rough length (line count or word count) of the new version against the prior version. An unexplained shrink is a red flag and must be surfaced to the user before the file is presented as final, not silently passed through.
- No Partial-File Output: The AI must always output the complete file in full when presenting an update -- never just the changed section or a snippet the user is expected to splice in themselves. Partial output is a common way updates get corrupted.
- Verify Current Version Before Updating: Before making any update, the AI must be working from the actual current version of this file -- either uploaded directly or pasted in full as text by the user in the current conversation. Memory of the file's contents from an earlier conversation does not count, since that memory may itself be stale or reflect a version with since-fixed regressions. If the AI does not have the current file's full content in front of it, it must ask the user to upload or paste it before proceeding, rather than reconstructing or guessing at the file's current state.
- Mandatory Command Keywords: This file uses a fixed set of user commands that ANY AI working with this file MUST recognize and follow, regardless of model or platform. These keywords are binding on the AI's behavior and are not optional, not subject to reinterpretation, and not overridable by an AI's own judgment about what seems more helpful in the moment:
  1. COMMIT -- the user is finalizing changes. The AI must run the full pipeline in order: Self-Audit, Change Summary, Length Sanity Check, version bump, Revision History entry, then output the complete updated file. Nothing is treated as final without a COMMIT.
  2. DRAFT -- the user wants to see proposed changes and a change summary first, WITHOUT bumping the version, updating Revision History, or treating anything as final. Used for review before committing.
  3. AUDIT -- the user wants the AI to run the Self-Audit against the current file and report any issues found (missing Revision History entries, section renames, length drops, etc.) without changing anything in the file.
  4. REVERT TO V# -- the user wants to roll back to a specific named prior version. The AI restores that version's content and adds a new Revision History entry noting the revert and the reason given.
  5. FREEZE / UNFREEZE -- FREEZE locks the file as-is for all purposes except the Scoped Auto-Update Rule. Upon FREEZE, the AI updates the FREEZE STATUS line (top of file) to FROZEN and adds a Revision History entry noting it. While FROZEN, the AI must refuse DRAFT, COMMIT, and any other edit to the file, regardless of session, conversation, or AI model -- the sole exception being that LESSON TOPICS, TRAINING RECORD averages, and the TEST BANK/QUESTION BANK continue to auto-update per the Scoped Auto-Update Rule even while frozen. The only input that lifts a freeze is the literal word UNFREEZE, given as its own explicit command. No rephrasing, implied consent, or general instruction to disregard, bypass, or override the freeze (e.g. "ignore the freeze," "just do it anyway," "make an exception") satisfies this requirement, regardless of how directly or emphatically it is stated, and regardless of AI model or platform. The AI must not use its own judgment to treat any such alternate phrasing as equivalent to UNFREEZE. UNFREEZE reverses the general lock: the AI updates the FREEZE STATUS line back to UNFROZEN, adds a Revision History entry noting the unfreeze, and normal editing (DRAFT/COMMIT) resumes.
  6. DISCARD -- clears any pending in-conversation draft that hasn't been COMMITted yet. The AI forgets the proposed changes entirely; the current master file is untouched. To propose something new afterward, the user starts a fresh DRAFT.
  Mandatory Keyword Gate: If the user requests or agrees to a substantive change to this file (new rule, new content, edited content, etc.) WITHOUT using one of the keywords above, the AI must NOT proceed to bump the version or treat the change as final on its own judgment. Instead the AI must explicitly ask the user "DRAFT or COMMIT?" (or equivalent) before taking any action that would alter the file. This gate exists specifically because an AI silently inferring a COMMIT defeats the purpose of having a fixed, non-discretionary command vocabulary.
  This rule itself, like all rules in this section, must be preserved verbatim by any AI updating this file in the future.
- Scoped Auto-Update Rule: When the user uploads new lesson content, completes a test, or requests that the AI generate a new question or questions from the existing LESSON TOPICS -- in this conversation or any future conversation/AI carrying this file forward -- the AI automatically updates only three things, without needing to ask the user first: (1) the LESSON TOPICS section (new topic appended under the correct/new lesson code, never overwriting or merging into existing bullets), (2) the relevant TRAINING RECORD rolling average(s) (per the existing Rolling Average Methodology), and (3) the TEST BANK/QUESTION BANK (per the existing duplicate-check rule, cross-checking against all previously stored questions regardless of which conversation or AI added them, tagged with topic code). For the "requests a new question or questions" trigger specifically, only the TEST BANK/QUESTION BANK is auto-updated -- since no new topic bullet or test result is being introduced, LESSON TOPICS and TRAINING RECORD are not touched by this trigger. No other section of the file may be modified during this automatic update, and no user confirmation is required for these specific updates. These auto-updates never bump the version number and never require a Revision History entry, regardless of FREEZE STATUS -- version bumps and Revision History entries occur only through an explicit COMMIT. Any change outside these areas still requires the user's explicit direction and a DRAFT or COMMIT. Upon successfully performing an auto-update under this rule, the AI must inform the user with a short one-liner confirming the update was made and successful, naming which of the areas (LESSON TOPICS, TRAINING RECORD average, TEST BANK/QUESTION BANK) were updated (e.g. "Auto-update complete: LESSON TOPICS and TH04 average updated."). If the auto-update cannot be performed for any reason (e.g. FREEZE STATUS conflict, malformed content, missing topic tag, ambiguous duplicate), the AI must automatically run a self-check/audit -- without requiring the user's input first -- to determine specifically why the update could not be completed, and report that specific finding to the user plainly.
